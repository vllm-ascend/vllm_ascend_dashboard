## 高优风险详情

### R-001: C++ 工具链兼容性（高）

**描述**：cppcheck 和 clang-tidy 需要在部署环境（Docker 容器）中安装并验证。不同版本的 cppcheck/clang-tidy 输出格式可能不同。

**影响**：C++ 安全指标（unsafe_functions、warning_suppression）无法采集，健康度评分缺少 C++ 安全维度。

**缓解措施**：
1. P1 阶段优先验证工具链可用性
2. 工具缺失时降级运行，collection_status 标记为 partial
3. Dockerfile 中预装 cppcheck 和 clang-tidy
4. 提供 cppcheck XML 输出解析的单元测试

### R-002: 采集性能（中）

**描述**：vllm-ascend 仓库有 4025 commits，csrc/ 目录 C++ 代码量大。全量执行 cloc + lizard + jscpd + ruff + mypy + cppcheck 可能耗时较长。

**影响**：采集任务超时，影响每日数据更新。

**缓解措施**：
1. 设置采集超时 15 分钟
2. 并行执行独立工具（cloc/lizard/jscpd 可并行）
3. 排除不需要的目录（node_modules/.git/build 等）
4. 监控采集耗时，记录 collection_duration

### R-003: lizard C++ 解析准确性（中）

**描述**：lizard 对 C++ 模板元编程、宏定义、条件编译的圈复杂度解析可能不完整。

**影响**：C++ 函数的圈复杂度数据偏低，影响健康度评分准确性。

**缓解措施**：
1. 与 cppcheck 的圈复杂度分析交叉验证
2. 记录 lizard 已知限制在文档中
3. 关注 lizard 版本更新

### R-007: C++ 危险函数误报（中）

**描述**：memcpy/strcpy 等函数在 C++ 算子代码中可能有合理使用场景（如性能优化的内存拷贝）。

**影响**：unsafe_functions_count 可能偏高，导致 C++ 安全维度评分偏低。

**缓解措施**：
1. 提供安全问题明细（CodeSecurityDetail），供人工审核
2. 支持管理员标记误报（后续迭代）
3. 文档中说明危险函数检测的局限性

### R-010: 健康度评分权重合理性（中）

**描述**：六维加权评分的权重（复杂度25%/重复率20%/规范20%/体量15%/技术债务10%/C++安全10%）是初步设定，可能需要根据实际数据调整。

**影响**：健康度评分可能不能准确反映代码质量。

**缓解措施**：
1. 初期将权重设为可配置项（ProjectDashboardConfig）
2. 收集 2-4 周数据后评估权重合理性
3. 参考 Cmetrics 的 adequacy 概念校准
