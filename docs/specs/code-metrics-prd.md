# 代码度量看板 — 产品需求文档（PRD）

> 版本：v1.0 | 日期：2026-07-09 | 作者：架构组

## 1. 项目背景与目标

### 1.1 背景

vllm-ascend 是针对华为昇腾 AI 处理器优化的 vLLM 项目，仓库地址 `vllm-project/vllm-ascend`。当前社区看板已具备 CI 看板、模型看板、性能看板、PR 流水线、测试可观测性等模块，但**缺少对代码本身质量的量化度量能力**。

vllm-ascend 仓库是一个**双语言栈**项目：
- **C++ 52.7%**（`csrc/` 目录 — 算子实现）
- **Python 42.0%**（`vllm_ascend/` 目录 — 插件代码）
- CMake 2.4%、Shell 2.3%、C 0.6%

现有看板已有 PR 级变更量（additions/deletions）、Commit 修改类型、CI 通过率等数据，但缺少代码规模、圈复杂度、重复率、安全规范等 Cmetrics 维度的度量。

### 1.2 目标

1. **建立代码质量基线**：每日采集 vllm-ascend 仓库的代码度量快照，形成可追溯的历史趋势
2. **覆盖双语言栈**：同时度量 Python（`vllm_ascend/`）和 C++（`csrc/`）代码，不遗漏 52.7% 的 C++ 代码
3. **对齐 Cmetrics 指标体系**：参考华为云 CodeCheck Cmetrics API，启用 C/C++ 专用指标（unsafe_functions、warning_suppression、huge_headerfile）
4. **支持版本对比**：支持按 tag/branch/commit 对比代码度量（如 v0.13.0 vs v0.18.0）
5. **可视化呈现**：提供 5 Tab 看板（总览、复杂度、重复率、热力图、趋势对比）

### 1.3 度量对象

**明确：度量对象是 vllm-ascend 仓库，不是看板自身的 backend/frontend 代码。**

| 模块 | 路径 | 语言 | 占比 |
|------|------|------|------|
| Python 插件 | `vllm_ascend/` | Python | 42.0% |
| C++ 算子 | `csrc/` | C++ | 52.7% |
| 测试 | `tests/` | Python | — |
| 性能测试 | `benchmarks/` | Python/Shell | — |
| 工具 | `tools/` | Python/Shell | — |
| 文档 | `docs/` | Markdown | — |
| CMake | `cmake/` + `CMakeLists.txt` | CMake | 2.4% |

## 2. 目标用户与使用场景

| 用户类型 | 使用场景 |
|---------|---------|
| 开发者 | 查看自己负责模块的复杂度趋势，识别需要重构的函数 |
| 维护者 | 评估 PR 合入对代码质量的影响，监控技术债务增长 |
| 架构师 | 对比版本间代码质量变化，制定重构优先级 |
| 管理层 | 通过健康度评分了解项目整体代码质量趋势 |

## 3. 功能需求清单

### P0 — 核心功能（必须）

| 编号 | 需求 | 说明 |
|------|------|------|
| F-001 | 代码规模采集 | 使用 cloc 统计 vllm-ascend 仓库全语言代码行数（Python/C++/CMake/Shell） |
| F-002 | 圈复杂度采集 | 使用 lizard 分析 Python + C++ 函数的圈复杂度、嵌套深度、函数行数 |
| F-003 | 代码重复率采集 | 使用 jscpd 检测全语言重复代码 |
| F-004 | 每日快照存储 | 每日定时采集并存储度量快照，支持幂等（同日期同分支不重复采集） |
| F-005 | 总览看板 | 健康度雷达图 + 核心指标卡片 + 语言/模块分布饼图 |
| F-006 | 复杂度详情 | 超大复杂度函数列表 + 复杂度分布直方图 |
| F-007 | 趋势对比 | 指标趋势折线图 + 版本间对比表 |

### P1 — 重要功能（应该）

| 编号 | 需求 | 说明 |
|------|------|------|
| F-008 | C++ 静态分析 | 使用 cppcheck + clang-tidy 分析 csrc/ 的安全问题 |
| F-009 | C++ 安全指标 | 启用 Cmetrics C/C++ 专用指标：unsafe_functions、warning_suppression、huge_headerfile |
| F-010 | Python Lint | 使用 ruff + mypy 分析 vllm_ascend/ 的规范问题 |
| F-011 | 技术债务统计 | 统计 TODO/FIXME/HACK 注释数及密度 |
| F-012 | 重复率详情 | 重复代码块明细列表 |
| F-013 | 文件热力图 | 从已有 Commit/PR 数据聚合文件变更频率 |
| F-014 | 健康度评分 | 六维加权评分（复杂度/重复率/规范/体量/技术债务/C++安全） |
| F-015 | 版本对比 | 支持按 tag 对比（如 v0.13.0 vs v0.18.0） |

### P2 — 增强功能（可以）

| 编号 | 需求 | 说明 |
|------|------|------|
| F-016 | 衍生指标 | 从已有 PR/Commit/CI 数据聚合（PR 大小分布、代码变更量、修改类型分布） |
| F-017 | 手动触发采集 | 管理员可手动触发指定日期/分支的采集 |
| F-018 | 数据导出 | 支持 CSV/JSON 导出度量数据 |
| F-019 | 告警集成 | 健康度降级告警、超大复杂度函数新增告警 |

## 4. 非功能需求

### 4.1 性能

| 指标 | 要求 |
|------|------|
| 单次采集耗时 | < 10 分钟（vllm-ascend 仓库约 4,025 commits） |
| API 响应时间 | < 500ms（P95） |
| 页面加载时间 | < 3 秒 |
| 采集内存占用 | < 2GB |

### 4.2 可用性

- 采集失败时保留上次成功快照，标注数据更新时间
- 工具链缺失时降级运行（如 cppcheck 未安装则跳过 C++ 安全分析），标注 `collection_status=partial`
- 支持手动重新采集覆盖错误数据

### 4.3 可维护性

- 复用现有 `GitHubLocalCache` 获取本地仓库路径，不新增仓库克隆逻辑
- 遵循现有模型/服务/API/调度器代码风格
- 采集工具链通过配置项控制启用/禁用

### 4.4 幂等性

- 同一日期 + 同一 repo + 同一 branch 的快照不重复采集
- 重新采集时先删除旧快照及关联明细，再写入新数据

## 5. 验收标准

### 5.1 采集验收

- [ ] 能正确采集 vllm-ascend 仓库 `vllm_ascend/` 和 `csrc/` 目录的代码行数
- [ ] 语言分布正确显示 Python/C++/CMake/Shell 四类
- [ ] 模块分布正确显示 vllm_ascend/csrc/tests/benchmarks/tools/docs 六类
- [ ] lizard 能同时分析 Python 和 C++ 函数
- [ ] cppcheck/clang-tidy 能分析 csrc/ 目录（工具安装时）
- [ ] 同日期同分支重复采集不产生重复记录

### 5.2 看板验收

- [ ] 总览页显示健康度雷达图、核心指标卡片、语言/模块分布
- [ ] 复杂度页显示分布直方图和超大函数列表
- [ ] 重复率页显示重复代码块明细
- [ ] 热力图页显示 Top 20 热点文件
- [ ] 趋势页支持选择指标和日期范围，显示折线图
- [ ] 支持选择两个 tag 进行版本对比

### 5.3 调度验收

- [ ] 每日定时采集任务正常执行
- [ ] 采集失败时记录错误日志，不影响其他任务
- [ ] 文件热力图每日更新
