# 版本质量评估技能

## 概述

通过数据驱动的结构化方法论评估开源项目的发布质量。收集多维指标（Bug 分布、测试覆盖、代码债务、社区健康、版本趋势），交叉分析揭示隐藏关联，产出客观的 HTML 质量评估报告，包含发布就绪判定和加固建议。

**核心原则：** 发布质量不是单一指标——它是 Bug 严重度、测试覆盖充分性、代码债务趋势和社区健康度的交集。一个版本可能 Bug 数量少，但如果测试覆盖盲区恰好是 Bug 高发区域，仍然不适合发布。

## 使用场景

- 发布新版本（RC 或稳定版）前评估
- 管理层询问"这个版本可以发布了吗？"
- Bug 数量增长，需要判断质量是否在恶化
- 评估功能是否应标记为稳定版还是实验版
- 规划发布加固冲刺，需要 P0/P1/P2 优先级

## 工作流

```
数据采集 → Bug 深度分析 → 测试覆盖分析
                          ↓
社区健康 ← 趋势对比 ← 代码质量分析
        ↓
发布质量评估 → 加固建议 → HTML 报告
```

### Phase 1: 数据采集

从多个来源并行采集原始数据：

| 来源 | 采集内容 |
|------|----------|
| GitHub Issues | 带 bug label 的 open issue，标题、标签、创建/关闭日期 |
| GitHub PRs | 已合并 PR 的时间戳、作者、标签（BugFix vs Feature） |
| Tag 对比 | 两个版本间的提交列表及分类（BugFix/Feature/Performance/Refactor/Doc/Test/CI/Misc） |
| CI 数据 | CI 运行通过率、失败趋势 |
| 模型支持矩阵 | 各模型的支持状态、特性列表 |
| 仓库统计 | Stars、Forks、open_issues_count |

### Phase 2: Bug 深度分析

#### 2.1 严重度分类

通过扫描标题 + 正文关键词对每个 Bug 分类：

| 严重度 | 关键词 | 含义 |
|--------|--------|------|
| 崩溃/卡死 | crash, hang, 卡死, 崩溃, killed | 服务不可用 |
| 输出错误 | 乱码, garbled, wrong, precision, accuracy, empty | 结果错误 |
| 启动失败 | failed, cannot, unable, 部署失败, ImportError | 无法部署 |
| 运行时错误 | RuntimeError, error code, AssertionError, TypeError | 执行错误 |
| 性能问题 | slow, 劣化, degrad, timeout, OOM | 低于预期 |

**关键洞察：** 计算崩溃+输出错误占比。如果 > 40%，版本存在生产事故级风险。

#### 2.2 根因模式分析

| 根因 | 如何检测 | 可控性 |
|------|----------|--------|
| 平台依赖 (CANN) | aclnn, socVersion | 外部依赖 |
| 算子缺陷 | operator name in error | 外部依赖 |
| 分布式/通信 | HCCL, allgather | 部分可控 |
| 精度/dtype | precision, bf16 | 可控 |
| 配置/参数 | port, key missing, shape mismatch | 可控 |

**关键洞察：** 计算可控 vs 外部依赖比例。如果 > 50% 可控，测试覆盖改进可直接减少 Bug。

### Phase 3: 测试覆盖分析

#### 3.1 测试指标

- 测试/bug 比率 = PR 测试数 / open bug 数。健康值 > 2.0。低于 0.5 表示测试跟不上 Bug 发现速度。
- CI 通过率趋势
- 各测试类型的覆盖情况（夜间测试、性能测试、E2E 测试）

#### 3.2 代码质量标记

| 标记 | 含义 | 风险 |
|------|------|------|
| HACK | 生产路径中的已知变通 | 多文件出现同一 HACK = 系统性问题 |
| FIXME | 已知未修复的 Bug | 通信/同步路径中高风险 |
| TODO | 待清理的临时代码 | 增长 = 债务积累 |

### Phase 4: 趋势对比

对比当前版本周期 vs 上一周期：

| 指标 | 当前 | 上一周期 | 趋势 |
|------|------|----------|------|
| 新增 open bug | ? | ? | ↑ = 恶化 |
| Bug 修复数 | ? | ? | — |
| Bug 净增长 | new - closed | — | 正值 = 恶化 |
| BugFix PR 合并数 | ? | ? | — |
| Feature PR 合并数 | ? | ? | Feature/BugFix > 1.5 = 扩展快于质量 |

同时识别**正向进展**：从实验到稳定的功能、新增测试覆盖、新增模型支持。

### Phase 5: 社区健康度

| 指标 | 健康范围 |
|------|----------|
| PR 合并中位时间 | < 24h |
| 24h 合并率 | > 80% |
| 活跃贡献者数 | > 30 |
| 贡献者多样性 | 无单一作者 > 20% |
| Stars/Forks | 增长趋势 |

### Phase 6: 发布质量评估

#### 模型成熟度矩阵

对每个支持的模型评估：bug 数、崩溃数、输出错误数 → 评定成熟度 → 给出判定（生产就绪 / 可用 / 基础可用 / 不可用）。

#### 功能稳定性矩阵

对每个功能评估：open bug 数、PR 测试数、测试/bug 比率 → 给出判定（稳定 / 基础 / 实验性）。

**稳定性门禁标准：**
- 稳定：测试/bug > 2.0，open bug < 5，有 PR E2E 覆盖
- 基础：测试/bug > 0.5，open bug < 10
- 实验性：测试/bug < 0.5 或零 PR E2E 覆盖

### Phase 7: 加固建议

按优先级给出建议：

| 优先级 | 标准 | 示例 |
|--------|------|------|
| P0 | 稳定功能中的崩溃/错误 bug；零覆盖关键场景 | 修复关键 bug；补充 E2E 测试 |
| P1 | 系统性技术债；高 LOC 模块缺 UT | 补充 UT；清理 HACK |
| P2 | 流程改进；长期质量门禁 | 模型适配门禁；变通跟踪 |

#### RC 收敛计划

| 阶段 | 标准 | 时间线 |
|------|------|--------|
| RC 发布 | P0 bug 清单确定 | 立即 |
| RC → 稳定 | P0 bug 清零 + 测试覆盖补齐 + 夜间通过率 > 80% 持续 7 天 | 2-3 周 |

## 报告结构

最终 HTML 报告应包含以下章节：

1. **Executive Summary** — 一表概览：版本判定、核心风险、正向进展、外部依赖风险、RC→稳定条件
2. **数据口径说明** — 每个指标的定义（什么算"bug"、"关键模块"、TODO 范围）
3. **趋势对比** — v(n-1) vs v(n) 指标 + 正向进展表
4. **Bug 分析** — 严重度分布、根因分析（含细分）、用户影响面
5. **测试覆盖** — CI 通过率、测试覆盖盲区、代码质量标记
6. **社区健康度** — PR 速度、贡献者、Bug 净增长
7. **发布评估** — 模型成熟度、功能稳定性、版本判定
8. **加固建议** — RC 收敛计划、P0 bug 修复排期、测试/代码/流程改进
9. **总结** — 平衡的结论（进展 + 风险）

## HTML 报告样式要求

生成的 HTML 报告必须采用**深色主题**，包含以下样式特征：

- 深色背景（`#0d1117`），卡片背景（`#161b22`），边框（`#30363d`）
- 文字主色（`#c9d1d9`），次要文字（`#8b949e`），强调色（`#58a6ff`）
- 严重度徽章：红/橙/黄/绿/蓝
- 统计卡片网格布局
- 关键发现使用 callout 提示框
- 表格带斑马纹
- 响应式布局，最大宽度 1150px
- 完整的 `<!DOCTYPE html>` 结构，可在任意浏览器渲染

## 常见错误

| 错误 | 修正 |
|------|------|
| 把所有 open issue 当 bug | 按 bug label 过滤，排除 RFC、文档、安装问题 |
| TODO 统计包含 tests/docs | 仅在核心代码目录 grep |
| 只展示问题 | 必须包含正向进展（相比上一版本改进了什么） |
| 无趋势数据 | 与上一版本周期对比——质量在改善还是恶化？ |
| 主观语言 | 使用"说明"列描述客观事实 |
| 缺少数据口径 | 明确定义每个指标的统计范围和方法 |
| 无 RC 收敛计划 | 给出具体 P0 bug 修复排期和预估工作量 |
